🔑 Key Design Principle: Separation of Concerns
  • Captus.sh: The "conductor" - orchestrates the workflow, manages dependencies, submits jobs
  • Functions_captus.sh: The "musicians" - contain the actual analysis code, can be called independently

2. Creating Your Own Package: Functions as Building Blocks

2.1 The Function Library Pattern

Each analysis step is encapsulated in its own function, making it:

  • Reusable: Call the same function with different parameters
  • Testable: Test each function independently
  • Maintainable: Update one function without breaking others
  • Portable: Functions can be copied to other projects

Example: Clean Function Structure

# In Functions_captus.sh: function captus_clean { INDIR=$1 # Input directory OUTDIR=$2 # Output directory SAMPLE=$3 # Sample name (from array task) # ANALYSIS CODE ls $INDIR/${SAMPLE}_R1.fq.gz # Verify input exists captus clean --reads $INDIR/${SAMPLE}_R1.fq.gz $INDIR/${SAMPLE}_R2.fq.gz \ --out $OUTDIR \ --overwrite }
💡 Function Design Tips:
  • Clear inputs: Use variables, not hardcoded paths
  • Single responsibility: Each function should do one thing (in case later you only want a certain task to be performed but not the other)
  • Document inputs/outputs: Comment what each parameter is for (to make your life easier when maintaining the code, or when other people use it)
  • Error checking: Verify inputs exist before processing/give yourself some instant feedback

Making Functions Callable from Outside

this is where we collect user inputs and pass to the actual functions
MY_FUNCTION=$1 #YOU don't need this line. just a reminder of what the first variable means METADATA=$2 #GET SAMPLE SAMPLE=$(sed -n "${SLURM_ARRAY_TASK_ID}p" ${METADATA} | cut -d ',' -f1) echo $SAMPLE #### # The "Router" - case statement at the end of Functions_captus.sh case "$1" in captus_clean) # $1 = function name, $2 = metadata, $3 = INDIR, $4 = OUTDIR, SAMPLE from array captus_clean $3 $4 $SAMPLE ;; captus_assembly) captus_assembly $3 $4 $SAMPLE ;; captus_extract) captus_extract $3 $4 $5 $SAMPLE # Note: $5 is BAITS file ;; captus_align) captus_align $3 $4 ;; IQTree) IQTree $3 $4 ;; *) echo "Usage: $0 {captus_clean|captus_assembly|...}" exit 1 ;; esac
before the router, we also capture some common variables:
  • MY_FUNCTION: captures the function name for clarity (not strictly needed but you can replace the $1 later with $MY_FUNCTION to remind yourself what the process is doing)
  • METADATA: captures the metadata file path (even if you might not need metadata everytime, it is easier to keep a consistent input structure)
  • SAMPLE: captures the sample name based on the array task ID (so you don't have to extract it every time in the function)
  • Router: the case statement at the end routes calls to the correct function based on the first argument
    • 💡 Router structure explained:
    • case "$1" in-- starts the case block, using the first command-line argument ($1) as the value to match.
    • pattern)-- each possible value is listed as a pattern ending with a )
    • ;;-- marks the end of a block.
    • *)-- wild card, match anything unmatched above. the default case if no patterns match.
    • esac-- closes the case block (case spelled backwards)..
    💡 Package Creation Pattern:
    1. Define functions: Each does ONE thing well
    2. Create router: case statement directs calls
    3. Export functions: Make them available to sbatch
    4. Document inputs: Comment what each parameter does

    3. SLURM Job Management

    3.1 Job Arrays: Parallel Processing

    What are Job Arrays?

    Job arrays allow you to submit a collection of jobs with a single sbatch command. Each sub-job has its own SLURM_ARRAY_TASK_ID environment variable.

    # In Captus.sh - Submitting an array job function captus_clean { JOB1=$(sbatch --array=1-4 \ # Run 4 parallel jobs (samples 1-4) --cpus-per-task=4 \ # Each job gets 4 CPUs --mem=4G \ # Each job gets 4GB memory --job-name="CLEANING" \ # Job name for tracking $SCRIPTS/Functions_captus.sh captus_clean $METADATA $DATA $RESULT1 \ | awk '{print $4}') # Capture JobID echo "Job $JOB1 submitted" }
    instead of specifying how the jobs are run in the script, you can specify it while submitting the job in commandline. This allows you to easily change the number of parallel jobs, or run on a subset of samples without changing the code.
    sbatch --array=${ARRAY} \ --cpus-per-task=2 \ --mem=2G \ --job-name="iqtree" \
    of course, you may not want to write all the sbatch parameters every time you submit a job. for everything that is common across all jobs, you can put it in the function script as a regular sbatch heading. we just want to change memory, cpus, and array size, name, and partition (short, medium, long) for different functions, so we specify those in the submission function in Captus.sh.
    #SBATCH --export=ALL #SBATCH --mail-user=your_rbge_email@rbge.org.uk #SBATCH --mail-type=END,FAIL #SBATCH --output $SCRATCH/slurm/slurm-%x-%A-%a.out #if you want to capture all slurm output in one place. optional, but you can do it

    3.2 Job Dependencies and Tracking

    Capturing Job IDs

    # Submit job and capture its ID JOB1=$(sbatch ... | awk '{print $4}') echo "Job $JOB1 submitted" # Use captured ID for dependencies JOB2=$(sbatch --dependency=afterok:$JOB1 ... | awk '{print $4}')

    Different Job Types in Workflow

    Job Type Example When to Use
    Array Jobs --array=1-4 Sample-level operations (cleaning, assembly, extraction)
    Single Jobs No array flag Cross-sample operations (alignment, concatenation)
    Dependent Jobs --dependency=afterok:$JOBID Pipeline steps that must wait for previous steps

    3.3 Resource Management

    # Each function specifies its own resource requirements function captus_clean { # Light resource usage sbatch --array=1-4 --cpus-per-task=4 --mem=4G ... } function captus_extract { # Heavier memory requirements sbatch --array=2,3 --cpus-per-task=4 --mem=16G ... } function captus_align { # Single job, but parallelized internally by captus sbatch --cpus-per-task=4 --mem=16G ... } function IQTree { # Many small jobs sbatch --array=1-152 --cpus-per-task=2 --mem=2G ... }
    ⚠️ Critical Considerations:
    • Memory per job: Array jobs each get the specified memory
    • CPU allocation: Total CPUs = array_size × cpus-per-task
    • Scratch space: Temporary files need independent directories in parallel jobs

    4. The Metadata-Driven Pattern

    📊 Metadata File Structure:
    # metadata_captus.csv (example) sample_id,species,location Sample1,Rhododendron sp.,SiteA Sample2,Rhododendron sp.,SiteB Sample3,Rhododendron sp.,SiteC Sample4,Rhododendron sp.,SiteD
    # In Functions_captus.sh - Accessing metadata SAMPLE=$(sed -n "${SLURM_ARRAY_TASK_ID}p" ${METADATA} | cut -d ',' -f1) # sed -n "${SLURM_ARRAY_TASK_ID}p" → Get line matching array ID # cut -d ',' -f1 → Extract first field (sample ID) # This pattern: # - Makes workflow sample-agnostic # - Easy to add/remove samples (just update metadata) # - easy to run on a subset of samples (e.g., --array=2,3)

    5. Temporary File Management in Parallel Jobs

    function captus_assembly { INDIR=$1 OUTDIR=$2 SAMPLE=$3 # Use sample-specific temp directory to avoid conflicts captus assemble --reads ... \ --tmp_dir $OUTDIR/${SAMPLE} \ # Unique per sample --overwrite # Clean up after yourself rm -rf $OUTDIR/${SAMPLE} # Remove temp files when done } function captus_extract { INDIR=$1 OUTDIR=$2 BAITS=$3 SAMPLE=$4 # Create temporary directory for this sample captus extract -a $OUTDIR/tmp_${SAMPLE} \ # Sample-specific temp # Clean up rm -rf $OUTDIR/tmp_${SAMPLE} $INDIR/${SAMPLE}__captus-asm/01_assembly/${SAMPLE}.fasta }
    🔧 Best Practice: Always use sample-specific temporary directories when running parallel jobs to prevent:
    • File write conflicts (two jobs writing to same file)
    • Accidental file deletion (one job removing another's files)
    • Resource contention (multiple jobs competing for same temp space)

    6. Complete Workflow Control

    Captus.sh - The Workflow Controller

    function main { # Setup setup_dir conda activate captus # STEP 0: Cleaning - Array job (4 samples) captus_clean # STEP 1: Assembly - Array job (depends on cleaning) # In production, you'd add: --dependency=afterok:$JOB1 captus_assemble # STEP 2: Extract - Array job (only samples 2,3) captus_extract # STEP 3: Align - Single job (parallelized internally) captus_align # STEP 4: Tree - Multiple array jobs (different partitions) IQTree $RESULT4/03_trimmed/06_informed/01_coding_NUC/02_NT $RESULT5/coding_NUC '1-152' IQTree $RESULT4/03_trimmed/06_informed/02_coding_PTD/02_NT $RESULT5/coding_PTD '1-52' IQTree $RESULT4/03_trimmed/06_informed/03_coding_MIT/02_NT $RESULT5/coding_MIT '1-34' } # Execution control - comment/uncomment steps as needed # captus_clean # captus_assemble # captus_extract # captus_align IQTree ... # Only run tree building

    🎮 Execution Control Patterns:

    • Comment/uncomment functions: Run specific workflow steps
    • Job dependencies: Ensure steps run in correct order
    • Array ranges: Process specific samples (e.g., --array=2,3)
    • Multiple arrays: Different partitions with different ranges

    7. Practical Exercise: Extending the Workflow

    Exercise: Add a New Analysis Step

    Task: Add a quality control step that runs FastQC on cleaned reads.

    # Step 1: Add function to Functions_captus.sh function run_fastqc { INDIR=$1 OUTDIR=$2 SAMPLE=$3 mkdir -p $OUTDIR/${SAMPLE} fastqc $INDIR/${SAMPLE}_R1.fq.gz $INDIR/${SAMPLE}_R2.fq.gz \ -o $OUTDIR/${SAMPLE} \ --threads 4 } # Step 2: Add router case case "$1" in run_fastqc) run_fastqc $3 $4 $SAMPLE ;; # ... existing cases esac # Step 3: Add submission function to Captus.sh function quality_control { JOB=$(sbatch --array=1-4 \ --cpus-per-task=4 \ --mem=4G \ --job-name="FASTQC" \ $SCRIPTS/Functions_captus.sh run_fastqc $METADATA $RESULT1 $RESULT1/fastqc \ | awk '{print $4}') echo "QC Job $JOB submitted" } # Step 4: Call it in workflow quality_control

    8. Best Practices Summary

    Category Best Practice Why
    Modularity One function = one analysis step Easy to test, debug, and reuse
    Separation Controller vs Worker files Clear responsibilities, easier maintenance
    Metadata Drive workflows with metadata files Scalable, reproducible, easy to update
    Resources Match resources to job requirements Efficient cluster usage, avoid over-requesting
    Cleanup Remove temporary files Prevent filling storage, avoid conflicts
    Documentation Comment function inputs and outputs Others (and future you) can understand the code

    🚨 Common Pitfalls to Avoid

    • Hardcoding paths: Use variables for all directories
    • Ignoring job dependencies: Jobs may start before inputs are ready
    • Shared temp files: Always use sample-specific temporary directories
    • No error checking: Always verify inputs exist before processing
    • Memory over-allocation: Array jobs multiply memory by array size

    📌 Key Takeaways

    • Two-file system provides clean separation of workflow logic and execution
    • Functions as building blocks create reusable, testable analysis components
    • SLURM arrays enable efficient parallel processing of multiple samples
    • Metadata-driven design makes workflows scalable and maintainable
    • Job ID tracking allows dependency management and workflow control
    • This pattern can be adapted for any bioinformatics pipeline, not just Captus
